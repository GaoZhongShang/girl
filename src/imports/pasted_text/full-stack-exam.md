## 一、用户管理模块（基础CRUD，面向企业员工管理）

### 1. 业务场景描述

某公司需要开发一款内部移动办公助手App，用于HR部门维护员工基础信息。HR管理员可通过App快速新增、查询、修改和删除员工数据，为后续考勤、薪资模块提供数据基础。要求数据录入规范、查询高效，并严格进行数据校验。

核心需求：管理员可通过移动端录入员工信息（姓名、年龄、邮箱），支持查看所有员工列表、根据员工ID查询单个员工详情、修改员工信息、删除离职员工信息；所有操作需经过数据校验，避免无效数据录入（如年龄超出范围、邮箱格式错误等）。

### 2. 功能要求（严格贴合技术栈）

**后端（Python Web框架 + MySQL）**

| 功能点         | 详细要求                                                     |
| :------------- | :----------------------------------------------------------- |
| 创建员工       | 接收姓名（非空，1~20字符）、年龄（18~60整数）、邮箱（合法格式）。使用框架的序列化/校验机制进行数据校验，校验通过后存入MySQL数据库，返回创建成功信息。 |
| 查询所有员工   | 查询MySQL中所有员工数据，按创建时间倒序返回，支持分页查询（可选）。 |
| 根据ID查询员工 | 根据员工ID从MySQL查询该员工完整信息，返回给前端，ID不存在时需返回明确的错误提示。 |
| 修改员工信息   | 接收修改后的数据，通过校验后更新MySQL中对应记录，返回修改成功信息。 |
| 删除员工       | 根据员工ID从MySQL删除对应记录，返回删除成功信息。            |

**前端（React Native）**

- 员工列表页：使用`FlatList`组件展示所有员工，每条记录包含姓名、年龄、邮箱及“查看详情”“删除”按钮，支持下拉刷新。
- 添加/编辑员工页：使用`TextInput`组件构建表单，配合表单校验规则（非空校验、格式校验）；编辑时自动填充现有员工数据。
- 调用后端API：使用`fetch`或`axios`库发送HTTP请求，使用`async/await`处理异步操作，展示加载状态（如`ActivityIndicator`）及错误提示（`Alert.alert`或Toast）。
- 删除确认：删除操作前弹出`Alert.alert`二次确认弹框，用户确认后才执行删除操作。
- 列表刷新：删除、添加、修改操作成功后，自动刷新员工列表。

### 3. 考察点（拆解细化，贴合实际开发）

**后端分层开发**

- 模块化设计：用户模块独立封装（Model、Controller/ViewSet、Service），确保代码可维护性。
- 分层结构清晰：Controller/ViewSet处理路由与请求映射，Service层封装核心CRUD逻辑，Model层定义数据实体与MySQL交互。
- 无论选择Flask、DRF还是Sanic，均应体现MVC或MV分层设计思想。

**数据校验**

- 使用框架内置的校验机制对姓名、年龄、邮箱进行严格校验（非空、格式、范围限制）。
- 校验失败返回明确的错误提示（状态码400），避免无效数据进入数据库。
- 掌握`Flask-RESTful`的`reqparse`参数解析与校验、DRF的`Serializer`字段级校验或Sanic的`webargs`参数校验库。

**基础增删改查**

- 后端与MySQL的交互（使用SQLAlchemy ORM或Django ORM），实现数据的新增、查询、修改、删除。
- 处理数据库连接异常和记录不存在等边界情况。
- ORM查询优化：按创建时间倒序排序、分页等。

**前后端交互**

- React Native与后端API的接口对接：`fetch`或`axios`发送GET/POST/PUT/DELETE请求。
- 前端表单提交、响应处理、错误提示的完整实现。
- 数据渲染的正确性：`FlatList`组件渲染列表，`useState`和`useEffect`管理组件状态。

## 二、登录 + JWT 身份验证（权限管控）

### 1. 业务场景描述

基于上述员工管理系统，新增权限管控功能：系统分为管理员和普通员工两种角色。只有管理员登录成功后，才能访问用户管理、设备管理等核心模块；普通员工仅可访问公开接口（如登录页面，个人资料查看等暂不实现）。登录时通过用户名和密码校验身份，校验通过后颁发JWT令牌，后续访问受保护接口时，需携带令牌进行身份验证，令牌过期或无效时拒绝访问并提示重新登录。

核心需求：实现管理员登录功能，生成合法JWT令牌；设计全局身份验证守卫，保护核心业务接口；令牌需设置合理过期时间，支持令牌解析、验证，确保系统安全性，防止未授权访问。

### 2. 功能要求（严格贴合技术栈）

**后端（Python Web框架 + MySQL）**

- 登录接口：接收用户名、密码。查询MySQL中管理员账号（预设1个：用户名admin，密码加密存储），校验用户名和密码（密码需与加密存储对比），校验通过后生成JWT令牌（包含管理员ID、角色信息，过期时间设为2小时），返回令牌给前端。
- JWT生成与解析：使用`PyJWT`库生成JWT令牌，设置密钥（JWT_SECRET，需配置为环境变量）、过期时间；提供令牌解析中间件或装饰器，从请求头中获取令牌，解析出管理员信息，验证令牌的有效性（是否过期、是否被篡改）。
- 守卫Guard实现：编写全局或路由级身份验证中间件/装饰器，拦截所有受保护接口，校验请求头中的`Authorization: Bearer `。若令牌无效、过期或不存在，返回未授权错误（状态码401）；若令牌有效，允许访问接口并传递管理员信息。
- 接口保护：明确划分受保护接口和公开接口（登录接口为公开接口），通过中间件或装饰器机制为受保护接口添加身份验证依赖，确保只有登录后的管理员才能访问。
- 密码安全：管理员密码需加密存储，推荐使用`bcrypt`或`werkzeug.security`生成哈希，校验时解密比对，避免明文存储带来的安全风险。

**前端（React Native）**

- 登录页：`TextInput`组件构建用户名/密码输入框，登录按钮，表单校验（非空）。
- 登录成功后，使用`AsyncStorage`（或`@react-native-async-storage/async-storage`）存储JWT令牌，并跳转到员工列表页（使用React Navigation进行页面导航）。
- 请求拦截器：封装API调用模块，在每次发送HTTP请求前，自动从`AsyncStorage`中读取JWT令牌，添加到请求头`Authorization: Bearer `中。
- 未授权（401）处理：若后端返回401状态码，自动清除本地存储的令牌，并使用React Navigation跳转回登录页面。
- 路由守卫：使用React Navigation的导航容器配合认证状态管理，构建认证栈（Auth Stack）和应用栈（App Stack），根据本地令牌是否存在决定展示哪个导航栈。

### 3. 考察点（拆解细化，贴合实际开发）

**Auth模块设计**

- 后端封装独立的Auth认证模块，整合登录逻辑、JWT生成与解析、守卫实现，模块划分清晰，与其他模块解耦。
- JWT_SECRET密钥需配置为环境变量，不得硬编码。

**JWT使用**

- `PyJWT`库的正确使用：JWT生成（payload结构、密钥设置、过期时间）、JWT解析（解码验证）、JWT验证核心逻辑，确保令牌安全性。
- 理解JWT的三部分结构（Header、Payload、Signature）及签名验证原理。

**身份验证守卫实现**

- 后端中间件/装饰器的正确使用，实现请求拦截、令牌校验，处理未授权场景，不影响正常业务逻辑。
- FastAPI用户可参考之前的JWT集成方案；Flask用户可使用`Flask-JWT-Extended`或基于`PyJWT`手动实现中间件。

**路由守卫与接口保护**

- 正确划分公开接口与受保护接口，通过依赖注入或中间件机制为受保护接口添加身份验证。
- 前端请求拦截器的使用：在axios实例的`interceptors.request`中自动为所有请求添加令牌；处理401响应的自动跳转。
- React Navigation认证流：根据令牌存在性自动切换导航栈，实现未登录不可访问受保护页面的效果。

**密码安全**

- 管理员密码加密存储，使用bcrypt或werkzeug.security等安全哈希算法。
- 登录时解密比对，避免明文存储带来的安全风险。

## 三、设备分类与设备管理（带关联查询）

### 1. 业务场景描述

公司内部需要管理办公设备（电脑、打印机、显示器等）。设备必须属于某个分类（如“IT设备”、“办公耗材”、“网络设备”）。管理员可管理分类，并查看某分类下的所有设备。要求每个设备必须属于一个分类，支持新增、修改、删除分类，查询分类列表，以及根据分类ID查询该分类下的所有设备，实现分类与设备的关联查询，确保数据的关联性和一致性。

核心需求：管理员可管理设备分类（添加分类名称、修改分类信息、删除分类，删除分类时需判断该分类下是否有设备，有设备则禁止删除，返回明确的错误提示）；管理设备（添加设备时选择所属分类，填写设备名称、型号）；支持根据分类ID查询该分类下的所有设备，前端渲染分类列表和对应设备列表。

### 2. 功能要求（严格贴合技术栈）

**分类模块后端**

- 添加分类：接收分类名称（非空、长度1~20字符），校验数据后存入MySQL分类表，返回添加成功信息。
- 修改分类：接收分类ID和修改后的分类名称，校验后更新MySQL分类表，返回修改成功信息，ID不存在时返回错误提示。
- 删除分类：删除前查询该分类下是否有设备，若有则返回错误提示（禁止删除），若无则删除分类，返回删除成功信息。
- 分类列表：查询所有分类，返回分类ID、分类名称、该分类下的设备数量。

**设备模块后端**

- 添加设备：接收设备名称（非空）、型号（可选）、所属分类ID（下拉选择，关联分类列表），校验后存入MySQL设备表（关联分类ID作为外键），返回添加成功信息。
- 添加设备时校验所选分类ID是否存在，避免无效分类关联。
- 关联查询：根据分类ID查询该分类下的所有设备，后端实现联表查询（分类表与设备表关联），返回设备列表（包含设备ID、设备名称、型号）。

**分类模块前端（React Native）**

- 分类列表页：展示所有分类（分类名称、设备数量），每条分类带“编辑”“删除”“查看设备”按钮。
- 添加/编辑分类：弹窗（Modal）或新页面表单，输入分类名称，提交后刷新分类列表。
- 删除分类：点击“删除”按钮弹出Alert确认弹框，确认后调用删除接口；若删除失败（分类下有设备），Alert显示后端返回的错误信息。

**设备模块前端（React Native）**

- 添加设备页面：表单包含设备名称输入框、型号输入框、所属分类下拉选择器（`Picker`或`ModalDropdown`）。下拉选项从后端分类列表接口动态获取。
- 设备列表页：进入页面时选择分类，或直接展示所有设备；支持按分类筛选设备。
- 分类设备关联视图：点击分类列表中的“查看设备”按钮，携带分类ID调用关联查询接口，跳转到该分类下的设备列表页面。

### 3. 考察点（拆解细化，贴合实际开发）

**ORM实体关联**

- 使用SQLAlchemy定义分类实体（Category）和设备实体（Device），实现OneToMany关联（一个分类对应多个设备）和ManyToOne关联（一个设备属于一个分类）。
- 正确设置外键：设备表分类ID字段关联分类表的ID，并设置级联删除规则（SET NULL或禁止级联删除）。
- Django ORM用户需掌握ForeignKey字段的用法及关联查询。

**联表查询**

- 后端通过SQLAlchemy的`join`方法或关联属性，实现分类与设备的联表查询，根据分类ID查询对应设备，确保查询高效，避免N+1查询问题。
- 掌握分类列表中“设备数量”字段的统计方法（`COUNT`聚合函数 + GROUP BY）。

**数据关联性校验**

- 删除分类时，校验该分类下是否有设备，避免出现“孤儿设备”（无归属分类的设备），确保数据一致性；若有设备，返回错误状态码（如400）并提示“分类下存在设备，无法删除”。
- 添加设备时，校验所选分类ID是否存在，避免无效分类关联（外键完整性约束）。

**前后端关联数据交互**

- 前端下拉选择分类时，从分类列表接口获取分类数据，动态渲染下拉选项。
- 关联查询时，前端携带分类ID请求后端，后端返回关联设备数据，前端渲染设备列表。
- 正确理解RESTful API中嵌套资源的表示方式：`/api/categories/{category_id}/devices` 或 `/api/devices?category_id={category_id}`。

## 四、API接口日志中间件（全局中间件）

### 1. 业务场景描述

为了便于系统运维和问题排查，需要实现全局接口日志功能，记录所有API接口的请求信息，包括请求方法、请求URL、请求IP、请求时间、响应状态码及响应耗时。日志需清晰、规范，便于后续查询和分析。要求日志功能不影响原有业务逻辑，无论接口请求成功还是失败，都能正常记录日志，且日志需输出到控制台和文件，便于开发调试和生产环境排查。

核心需求：实现全局中间件，拦截所有前端发起的API请求，在请求到达业务接口前记录请求信息，在业务处理完成后记录响应状态和耗时；中间件不干扰接口的正常执行，不修改请求参数和响应数据；日志格式统一，包含关键信息，支持日志轮转和分级存储。

### 2. 功能要求（严格贴合技术栈）

**后端全局日志中间件**

- 全局中间件实现：使用对应Python Web框架的中间件机制，编写全局请求日志中间件（LoggingMiddleware），应用到所有路由，拦截所有HTTP请求。
- 日志记录内容：每次请求API时，日志应包含以下信息：
  - 请求时间：格式为`YYYY-MM-DD HH:MM:SS`（如2026-05-21 14:30:00）；
  - 请求方法：GET / POST / PUT / DELETE等；
  - 请求URL：完整的API路径（如/api/users、/api/categories）；
  - 请求IP：发起请求的客户端IP地址（需考虑代理场景，正确获取真实IP）；
  - 响应状态码：接口返回的HTTP状态码（200、400、401、500等）；
  - 响应耗时：从请求开始到响应结束的时间差（毫秒或微秒）。
- 不影响业务逻辑：中间件仅负责记录日志，不修改请求参数、不拦截响应数据，无论接口请求成功、失败（如参数错误、未授权、系统异常），都能正常记录日志。
- 全局应用：将日志中间件注册为全局中间件，应用到框架的所有路由，无需为每个接口单独配置，实现一次配置、全局生效。
- 日志输出：同时输出到控制台（便于开发调试）和日志文件（便于生产环境排查），使用Python标准库`logging`模块实现日志分级和文件轮转（`RotatingFileHandler`）。

**前端日志上报（加分项）**

- 前端可考虑添加客户端日志收集机制，上报App运行时的异常和用户行为日志（可选实现）。

### 3. 考察点（拆解细化，贴合实际开发）

**Web框架中间件使用**

- 掌握对应Python Web框架中间件的编写方法，理解中间件的执行流程（请求到达前、请求处理后）。
- FastAPI用户需掌握`@app.middleware("http")`的编写和使用。
- Flask用户需掌握`@app.before_request`、`@app.after_request`装饰器或Flask应用中间件机制。
- Sanic用户需掌握`@app.middleware('request')`和`@app.middleware('response')`装饰器的使用。
- DRF用户可通过自定义中间件类或使用Django的中间件系统。

**请求对象（Request）使用**

- 从框架的Request对象中获取请求方法、URL路径、客户端IP地址等信息，熟练操作请求对象的属性和方法。

**日志格式规范与日志模块使用**

- 日志内容完整、格式统一，包含全部关键信息，便于后续排查问题。
- 掌握Python `logging`模块的基本使用：日志级别（DEBUG、INFO、WARNING、ERROR）、日志格式化（`Formatter`）、Handler配置（`StreamHandler`、`RotatingFileHandler`）及Logger对象的获取。

**中间件与业务解耦**

- 确保中间件仅负责日志记录，不侵入业务逻辑，不影响接口的正常请求和响应，体现分层设计和关注点分离思想。
- 正确计算接口响应耗时：在请求到达时记录开始时间（`time.time()`或`datetime.now()`），在响应发送前计算耗时并追加到日志中。

## 五、统一接口响应格式 + 全局异常拦截

### 1. 业务场景描述

为了规范前后端API交互，提升接口的可维护性和易用性，需要统一所有API接口的响应格式，无论接口请求成功还是失败，都返回固定结构的JSON数据，便于前端统一处理响应、展示提示信息。同时，需要实现全局异常拦截，捕获系统中所有未处理的异常（如参数校验错误、数据库异常、未授权异常、接口不存在等），统一返回标准错误格式，避免直接返回系统原生错误信息（防止泄露系统细节，提升用户体验）。

核心需求：成功响应统一包装为指定JSON结构，包含状态码、成功提示、返回数据；失败响应由全局异常过滤器捕获，统一包装为指定JSON结构，包含错误状态码、错误提示、错误详情；确保所有接口（包括异常场景）都遵循该响应格式，前端可根据状态码和提示信息统一处理交互逻辑。

### 2. 功能要求（严格贴合技术栈）

**统一接口响应格式**

- 所有API接口返回的JSON数据，严格遵循以下标准结构：

| 字段      | 类型                  | 说明                                           | 示例                       |
| :-------- | :-------------------- | :--------------------------------------------- | :------------------------- |
| `code`    | int                   | 状态码：200表示成功，非200表示失败             | 200、400、401、500         |
| `message` | str                   | 提示信息：成功时为"成功"，失败时为具体错误提示 | "成功"、"邮箱格式错误"     |
| `data`    | object / array / null | 数据：成功时返回具体业务数据，失败时返回null   | `{"users": [...]}`、`null` |

**成功响应处理**

- 后端通过框架提供的拦截器或响应封装工具，对所有成功响应进行统一包装，无需在每个接口函数中手动封装响应格式，减少冗余代码。
- DRF用户可使用`drf-standardized-responses`第三方库实现标准响应格式的统一封装。
- 非DRF框架用户可自行封装响应函数或编写响应中间件。

**全局异常拦截**

- 编写全局异常处理器（Exception Handler），捕获系统中所有未处理的异常，包括：
  - 参数校验异常（如Pydantic校验失败、Serializer校验失败）
  - 数据库异常（如连接失败、SQL语法错误、唯一约束冲突）
  - 未授权异常（JWT令牌无效、过期）
  - 接口不存在异常（404 Not Found）
  - 自定义业务异常（如“分类下有设备，禁止删除”）
- 针对不同类型的异常，设置对应的状态码和错误提示：

| 异常类型     | 状态码(code) | 错误提示(message)              |
| :----------- | :----------- | :----------------------------- |
| 参数校验失败 | 400          | "参数校验失败：[具体字段错误]" |
| 资源不存在   | 404          | "[资源名称]不存在"             |
| 未授权访问   | 401          | "未授权访问，请先登录"         |
| 权限不足     | 403          | "权限不足，无法执行此操作"     |
| 业务冲突     | 409          | "业务冲突：[具体原因]"         |
| 系统异常     | 500          | "系统异常，请稍后重试"         |

- 异常捕获后，统一包装为标准响应格式返回给前端，避免返回框架原生的错误信息或HTML页面。
- 自定义业务异常：支持开发者定义自定义异常类，抛出后被全局异常处理器捕获并返回对应的错误提示和状态码，体现良好的错误处理设计。

**前端统一响应处理**

- 封装统一的API请求模块，根据响应中的`code`字段统一处理：
  - `code === 200`：正常返回`data`数据，业务正常处理。
  - `code !== 200`：根据`message`字段向用户展示错误提示（Alert或Toast）。
  - 特定状态码处理：如`code === 401`时自动清除本地令牌并跳转登录页。

### 3. 考察点（拆解细化，贴合实际开发）

**拦截器/响应封装使用**

- 掌握对应框架的响应封装机制或第三方库的使用，实现对成功响应的统一包装，减少代码冗余，规范API输出。
- 能够自定义成功响应函数，设计统一的返回格式。

**全局异常处理器实现**

- 编写全局异常处理器，捕获各类异常（包括框架原生异常和自定义异常），区分异常类型，设置对应的状态码和错误提示。
- DRF用户需自定义`exception_handler`并配置在`REST_FRAMEWORK`的`EXCEPTION_HANDLER`设置中。
- Flask用户可使用`@app.errorhandler`装饰器自定义各类异常处理。
- Sanic用户可使用`@app.exception`装饰器注册全局异常处理器。
- 能够处理好“异常未捕获导致系统崩溃”的场景，确保API接口始终返回标准格式的JSON响应。

**接口规范意识**

- 理解统一接口响应格式的重要性，能够设计合理的状态码和提示信息，便于前后端交互和问题排查。
- 前后端分离开发中，统一的响应格式是团队协作的基础规范。

**异常处理逻辑**

- 能够处理常见的业务异常和系统异常，避免异常信息泄露系统敏感细节，提升API的安全性和用户体验。
- 掌握自定义业务异常的设计与使用，体现良好的面向对象设计能力。
- 了解日志记录与异常处理的结合使用：全局异常处理器捕获异常后，除返回标准错误响应外，还应记录错误日志以便后续排查。

## 补充说明

### 1. 技术栈约束

| 层级           | 技术选型                                               | 约束说明                                                   |
| :------------- | :----------------------------------------------------- | :--------------------------------------------------------- |
| 前端           | React Native（Expo或React Native CLI）                 | 必须使用React Native 0.72+版本，支持iOS和Android双平台开发 |
| 后端Python框架 | **二选一**：Flask + Flask-RESTful/Flask-Smorest、Sanic | 任选其一完成，ORM统一使用SQLAlchemy                        |
| 数据库         | MySQL 8.0+                                             | 存储所有业务数据，使用InnoDB引擎以支持外键约束和事务       |
| 数据库ORM      | SQLAlchemy（Flask/Sanic选型）或Django ORM（DRF选型）   | 禁止使用原生SQL拼接，必须使用ORM进行数据操作               |
| HTTP客户端     | axios 或 React Native内置`fetch`                       | 前端与后端交互统一使用，禁止混用多种请求库                 |

### 2. 代码规范

- **后端**：遵循分层开发思想（Controller/ViewSet层、Service/Business层、Model/DAO层、DTO/Serializer层），代码需结构清晰、注释完整。
- **前端**：遵循React Native组件化开发规范，使用函数式组件和React Hooks（`useState`、`useEffect`、`useContext`等），命名规范统一（如变量采用camelCase，常量采用UPPER_SNAKE_CASE）。
- **API路径规范**：遵循RESTful设计原则，使用名词复数表示资源（如`/api/users`、`/api/categories`、`/api/devices`），使用HTTP方法语义（GET查询、POST创建、PUT修改、DELETE删除）。
- **错误处理和日志**：关键操作需添加try-catch和日志记录，确保代码健壮性。
- **环境配置**：所有敏感配置（数据库连接字符串、JWT_SECRET等）必须通过环境变量管理，严禁硬编码。

### 3. 可运行性

所有功能实现后，需确保满足以下可运行要求：

- 前端React Native App可正常编译运行（iOS Simulator/Android Emulator/真机均可）。
- 后端API服务可正常启动，所有接口可正常请求响应。
- MySQL数据库服务可正常连接，数据可正常读写。
- 各模块业务流程完整无报错、无逻辑漏洞、无内存泄露（如未清理的定时器、未取消的请求等）。
